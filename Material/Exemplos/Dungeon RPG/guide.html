CAPÍTULO 01

Arquitetura Geral

Por que um arquivo só, e como esse arquivo se organiza por dentro.

Dungeon Quest é um jogo de ação 2D visto de cima (estilo Zelda clássico), inteiramente contido em index.html: um único arquivo com <style>, <body> (o HUD e os overlays de UI) e um <script> com o motor do jogo inteiro. Não há bundler, não há módulos, não há dependências externas além de uma fonte do Google Fonts para o título.

Essa escolha não é ingenuidade — é uma decisão deliberada de portabilidade: abrir o arquivo em qualquer navegador é o suficiente para jogar. O preço dessa escolha é organização disciplinada dentro de um espaço de nomes global, e é exatamente essa disciplina que este guia vai desmontar.

As três camadas do arquivo

<style>→CSS do HUD, telas de overlay (mapa-múndi, save, diálogo) e da tela de título
<body>→<canvas id="gameCanvas">, painel de HUD lateral, minimapa e overlays DOM
<script>→Constantes → Tiles/Mapas → Classes de entidade → Sistemas → class Game → boot

O laço principal (game loop)

No fim do arquivo, class Game é instanciada e inicia um loop clássico de requestAnimationFrame com delta time em segundos, limitado a 50ms para evitar "saltos" quando a aba perde foco:

_loop(ts){
  const dt = Math.min((ts - (this._lastTime ?? ts)) / 1000, 0.05);
  this._lastTime = ts;
  this._update(dt);
  this._draw();
  requestAnimationFrame(ts2 => this._loop(ts2));
}
class Game._loop · linha ~22586

_update(dt) é um grande roteador de estado: primeiro atualiza o input, depois checa — em ordem de prioridade — se alguma tela modal está ativa (tela de save, tela de personagem, de equipamento, de armas, bestiário, diálogo, loja...). Se estiver, delega o update para aquela tela e sai cedo (return). Só quando nenhum overlay está ativo o update "cai" para a simulação normal do mundo: player, câmera, inimigos, projéteis, partículas.

Padrão a observar Esse "early return em cascata" é o mecanismo central de gerência de estado do jogo inteiro. Não existe uma máquina de estados formal (enum + switch) — existe uma sequência de if (tela.active) { tela.update(); return; }. Simples, mas eficaz para o tamanho do projeto, e é o padrão que qualquer nova tela modal deve seguir.

Desenho: _draw()

Espelhando o update, _draw() desenha o mundo (tiles → objetos → itens → NPCs → inimigos → player → projéteis → partículas, nessa ordem de profundidade), aplica a escala de câmera via ctx.scale, desenha o fade de transição de mapa por cima, e finalmente desenha a tela modal ativa (se houver) por cima de tudo. O HUD lateral não é desenhado no canvas — é DOM/CSS real, atualizado por chamadas pontuais como this.hud.update(...).

CAPÍTULO 02

Tiles e Mapas

Como o mundo é descrito em dados, não em código.

Todo mapa do jogo é uma matriz 2D de números. Um tile é um inteiro; um mapa é uma matriz desses inteiros; e um dicionário central traduz cada inteiro em cor, colisão e comportamento. Essa indireção — número → significado — é o que permite ter dezenas de mapas sem escrever uma linha de lógica de desenho para cada um.

O dicionário T

A constante T nomeia cada tipo de tile com uma constante inteira, para que o resto do código nunca precise usar números mágicos:

Todas as coordenadas de conteúdo (tx/ty) são em tiles, não pixels — a conversão para pixels (tx * TILE_SIZE) acontece só na hora de instanciar o objeto real no mapa. Isso deixa o arquivo de definição legível e fácil de desenhar à mão em uma grade mental.

Regra de abertura de borda O próprio código documenta essa regra em comentário: o tile na borda do mapa (linha 0, última linha, coluna 0, última coluna) correspondente a uma direção de connections precisa ser não-sólido. É assim que o jogador consegue fisicamente alcançar aquela célula de borda para a troca de mapa ser detectada — se a borda for parede, a conexão nunca dispara.

CAPÍTULO 03

Navegação entre Mapas

ActiveMap, MapManager e o mapa-múndi — e um bug real de coordenadas negativas.

Três classes dividem a responsabilidade de "onde o jogador está": ActiveMap é o mapa carregado agora (tiles + entidades vivas), MapManager troca de um ActiveMap para outro com fade e preserva estado entre visitas, e WorldMap desenha a visão aérea de todos os mapas conectados (tecla M).

Troca de mapa com fade

MapManager não troca de mapa instantaneamente — ele anima uma cortina preta no próprio canvas (não é um elemento DOM) com dois estágios: escurece, executa a troca real no meio do escurecimento (_fadeMidCb), depois clareia:

updateFade(dt){
  if(this._fadeDir===0) return;
  if(this._fadeDir===1){
    this._fadeAlpha = Math.min(1, this._fadeAlpha + dt/0.22);
    if(this._fadeAlpha>=1 && !this._fadeMidDone){
      this._fadeMidDone = true;
      this._fadeMidCb?.();     // troca o mapa de verdade aqui, na tela preta
      this._fadeDir = -1;      // agora clareia
    }
  } else {
    this._fadeAlpha = Math.max(0, this._fadeAlpha - dt/0.28);
    if(this._fadeAlpha<=0){ this._fadeDir=0; this._fadeCb?.(); this._fadeCb=null; }
  }
}
MapManager.updateFade · linha ~21301

Trocar o mapa "no escuro" evita que o jogador veja o pop-in de tiles e entidades sendo recriados — um truque simples de mascaramento visual bastante comum em jogos 2D.

Estado por mapa persiste entre visitas

Antes de trocar de mapa, MapManager._saveMapState() tira uma "foto" do mapa atual: itens já coletados, baús abertos, tiles quebráveis destruídos, estado de interruptores e itens dropados no chão. Essa foto fica em this.mapStates[mapId] — um objeto vivo em memória, separado do SaveSystem (que persiste em localStorage). Isso é o que garante que, ao sair da vila e voltar, o baú que você já abriu continue aberto.

O mapa-múndi e o bug de coordenadas negativas

A visão aérea (WorldMap) posiciona cada mapa numa grade lógica {gx, gy} definida em WORLD_LAYOUT. O problema: conforme o mundo cresceu para novas direções (mapas ao norte e a oeste da vila inicial), algumas células passaram a ter gx ou gy negativos — e um <canvas> não tem coordenadas negativas utilizáveis diretamente sem cortar conteúdo.

A correção foi calcular um deslocamento (offset) a partir do menor gx/gy de todo o layout, e usar esse offset em toda conversão de grade → pixel:

_rebuildLayout() {
  const gxs = Object.values(WORLD_LAYOUT).map(v => v.gx);
  const gys = Object.values(WORLD_LAYOUT).map(v => v.gy);
  // Guarda o menor gx/gy como offset — permite coordenadas negativas
  // (ex: mapas ao norte/oeste do "mapa 0,0" original) sem cortar o canvas.
  this._minGx = Math.min(...gxs);
  this._minGy = Math.min(...gys);
  const maxGx = Math.max(...gxs);
  const maxGy = Math.max(...gys);
  this._totalW = (maxGx - this._minGx + 1) * this.CELL_W + this.PAD_X * 2;
  this._totalH = (maxGy - this._minGy + 1) * this.CELL_H + this.PAD_Y * 2;
  this._canvas.width  = this._totalW;
  this._canvas.height = this._totalH;
}

_cellRect(mapId) {
  const pos = WORLD_LAYOUT[mapId];
  if (!pos) return null;
  return {
    x: this.PAD_X + (pos.gx - this._minGx) * this.CELL_W,
    y: this.PAD_Y + (pos.gy - this._minGy) * this.CELL_H,
    w: this.CELL_W - 14, h: this.CELL_H - 14,
  };
}
class WorldMap · linha ~21997
Lição de arquitetura Sempre que uma coordenada "lógica" (grade do mundo, índice de inventário, id de slot) pode variar livremente — inclusive para negativo — e precisa virar uma coordenada "física" (pixel, índice de array), vale a pena calcular o mínimo do conjunto de dados antes de desenhar, em vez de assumir que tudo começa em zero. É exatamente esse tipo de suposição implícita que quebra silenciosamente quando o conteúdo cresce em uma direção não prevista.

Como uma troca de mapa é disparada

O gatilho é puramente geométrico: quando o jogador atinge a última linha/coluna válida de tile em uma direção que tem entrada em connections, o MapManager inicia o fade e, no callback do meio, chama load(nextMapId, player, spawnOverride) — que reposiciona o player no spawnX/spawnY do mapa de destino, já convertido de tile para pixel e centralizado na célula.

Câmera e Entrada

Duas classes pequenas que resolvem dois problemas clássicos de jogos em canvas.

InputManager: teclas com estado de "acabou de apertar"

Jogos precisam distinguir "a tecla está pressionada agora" (útil para movimento contínuo) de "a tecla foi pressionada neste frame" (útil para ações de disparo único, como atacar ou abrir um menu). O InputManager resolve isso com dois mapas — keys (estado contínuo) e um buffer que vira justPressed a cada frame:

update() {
  this.justPressed = this._pressBuffer;
  this._pressBuffer = {};
}
isDown(code) { return !!this.keys[code]; }
isJust(code) { return !!this.justPressed[code]; }

Camera: seguir com suavização, sem sair do mapa

A câmera não gruda direto no jogador — ela persegue um alvo com interpolação exponencial (lerp), o que produz aquele leve "atraso" suave característico de RPGs top-down:

follow(ex, ey, mapW, mapH) {
  const vw = viewW(), vh = viewH();
  const tx = ex - vw/2, ty = ey - vh/2;
  this.x += (tx - this.x) * 0.12;   // 12% da distância a cada frame
  this.y += (ty - this.y) * 0.12;
  this.x = Math.max(0, Math.min(Math.max(0, mapW - vw), this.x));
  this.y = Math.max(0, Math.min(Math.max(0, mapH - vh), this.y));
  this._updateOffset();
}
class Camera.follow · linha ~1729

O clamp com Math.max(0, Math.min(...)) é o que impede a câmera de mostrar área fora do mapa nas bordas — ela trava assim que a janela de visão encosta na borda do mundo. snapTo() é a mesma lógica sem interpolação, usada ao trocar de mapa (a câmera não deveria "voar" suavemente de um mapa para o outro).

Detalhe sutil _updateOffset() arredonda a posição da câmera para o pixel mais próximo em espaço de tela (Math.round(this.x * GAME_SCALE) / GAME_SCALE) antes de desenhar. Isso evita o "sub-pixel jitter" — tremulação visual que aparece quando sprites com image-rendering: pixelated são desenhados em posições fracionárias.

Entidades e o Jogador

Uma base pequena e comum sob o jogador, os inimigos e os NPCs.

A classe Entity: o mínimo que toda coisa viva precisa

No topo da hierarquia está Entity — não uma classe abstrata formal, mas uma base concreta com o que qualquer coisa com posição, vida e animação precisa:

class Entity {
  constructor(x, y, w, h) {
    this.x=x; this.y=y; this.w=w; this.h=h;
    this.vx=0; this.vy=0; this.hp=1; this.maxHp=1;
    this.alive=true; this.facing='south';
    this.animFrame=0; this.animTimer=0; this.animSpeed=0.12;
    this.invincible=0; this.knockback={x:0,y:0,t:0};
    this.stunTimer=0;
  }
  get cx(){ return this.x+this.w/2; }
  get cy(){ return this.y+this.h/2; }

  hit(dmg, srcX, srcY) {
    if(this.invincible>0) return false;
    this.hp-=dmg; this.invincible=0.8;
    const dx=this.cx-srcX, dy=this.cy-srcY, d=Math.sqrt(dx*dx+dy*dy)||1;
    this.applyKnockback(dx/d, dy/d, 120);
    if(this.hp<=0) this.alive=false;
    this._spawnDamageText(dmg);
    return true;
  }
}
class Entity · linha ~3022

Player, Enemy e todas as suas subclasses (Goblin, Mage, os oito chefes, etc.) estendem Entity. O que Entity fornece de graça a qualquer descendente:

  • Centro geométrico (cx/cy) calculado a partir de canto + largura/altura — usado para distância, mira e colisão circular aproximada.
  • Dano e i-frames: hit() ignora golpes durante invincible > 0, aplica knockback proporcional à direção do golpe, e sinaliza morte via alive = false — nunca removendo o objeto na hora, o que evita mutar arrays durante a iteração.
  • Texto de dano flutuante: _spawnDamageText() decide a cor (vermelho para o jogador, amarelo para inimigos) checando this instanceof Player.
  • Timers genéricos: updateTimers() decai invencibilidade e aplica o vetor de knockback ao longo do tempo; updateAnim() avança o frame de sprite só quando moving é verdadeiro.
Por que isso importa Como hit(), o knockback e a invencibilidade vivem em Entity e não em cada subclasse, todo objeto do jogo — de um Goblin a um AngelBoss de três fases — ganha essas regras de combate automaticamente. Uma nova classe de inimigo praticamente nunca precisa reimplementar "levar dano".

Defesa em camadas no hit() do jogador

Player sobrescreve hit() de Entity para inserir três camadas de mitigação antes de chamar super.hit(): primeiro o Escudo Arcano (bloqueio total, consome carga), depois um amuleto temporário que reduz 1 de dano, depois a defesa plana da armadura equipada — só o resíduo final chega ao hit() base, que desconta HP.

Inimigos e IA

Uma base comum, um punhado de padrões de comportamento, e chefes que os combinam.

O jogo tem mais de 30 tipos de inimigos e chefes — de um slime simples a um Anjo Caído com três fases de combate. Nenhum deles é escrito do zero: todos estendem Enemy (que por sua vez estende Entity), e a maior parte da variedade percebida vem de combinar um pequeno conjunto de padrões de IA reutilizáveis, não de reinventar a lógica de movimento a cada inimigo novo.

Enemy: estatísticas por tipo, comportamento comum

A base de todo inimigo simples guarda as estatísticas de cada type em uma tabela local, e resolve a instância corrente por chave:

    const total = this.dropTable.reduce((s, e) => s + e.w, 0);
    let r = Math.random() * total;

NPCs, Missões e Diálogo

Três classes de status por missão, e um roteador que decide o que dizer.

Estados de uma missão: três, sem mais

Toda a lógica de missões gira em torno de uma máquina de três estados guardada em QuestSystem.progress[questId]: active (aceita, em progresso), ready (concluída, aguardando entrega) e done (entregue). Não existe um quarto estado "disponível" — isso é derivado, não armazenado: uma missão sem entrada em progress ainda não foi oferecida.

// progress[questId] = { status: 'active'|'ready'|'done', count: N }

interact(giverDef) {
  const ready   = giverDef.quests.filter(qid => this.progress[qid]?.status === 'ready');
  const pending = giverDef.quests.filter(qid => !this.progress[qid]);
  const active  = giverDef.quests.filter(qid => this.progress[qid]?.status === 'active');

  if (ready.length   > 0) return { action: 'deliver',  questId: ready[0]   };
  if (pending.length > 0) return { action: 'offer',    questId: pending[0] };
  if (active.length  > 0) return { action: 'progress', questId: active[0] };
  return { action: 'none' };
}

Progresso de missão é orientado a eventos

O sistema não faz polling do estado do mundo — ele expõe métodos de evento (onKill, onCollect, onExplore) que o resto do jogo chama no momento exato em que algo acontece (um inimigo morre, um item é coletado, um mapa novo é visitado). Cada método varre QUEST_DEFS procurando missões ativas cujo type/target combinem com o evento:

Interface: HUD e Telas

Um HUD que é DOM de verdade, e telas modais que são canvas de verdade.

Duas tecnologias de UI, uma decisão consciente

O jogo usa duas abordagens de interface lado a lado, e a escolha entre elas não é acidental:

  • HUD permanente (corações, gemas, chaves, contadores de munição) é HTML/CSS real — os elementos #hud-panel, #hud-hp-row etc. definidos no <style> do topo do arquivo. A classe HUD apenas escreve/atualiza esse DOM a cada frame relevante.
  • Telas modais (personagem, equipamento, armas, bestiário, tela de título, game over) são desenhadas dentro do canvas, com ctx.fillRect/fillText manuais.

A lógica por trás da divisão: o HUD precisa ficar sempre legível e nunca "por cima" da ação, então layout CSS real (flexbox, responsivo) resolve melhor. Já as telas modais fazem parte da cena do jogo — precisam se misturar visualmente com o mundo (fundo semitransparente sobre o gameplay pausado) e são mais simples de manter consistentes com a estética pixelada desenhando-as com os mesmos primitivos do resto do jogo.

O padrão comum de toda tela modal

Apesar de conteúdos muito diferentes, CharacterScreen, EquipmentScreen, WeaponScreen e CodexScreen compartilham a mesma forma:

  • Uma flag active que o roteador de Game._update() checa em cascata (capítulo 1).
  • Um _alpha que anima de 0 a 1 na abertura, para um fade suave em vez de um "pop" abrupto.
  • update(dt, input) lê teclado (setas para navegar, Esc/tecla própria para fechar) e draw(ctx) desenha o painel por cima de tudo.

Esse padrão é o que torna o roteador do capítulo 1 tão simples de estender: uma tela nova só precisa seguir o mesmo contrato (active/update/draw) para se encaixar na cascata de _update() sem tocar em nenhuma outra tela existente.

Persistência e Condição de Vitória

Como um objeto cheio de instâncias de classe vira JSON — e volta.

O problema central de qualquer save system

player.inventory guarda objetos com uma função onUse anexada (capítulo 9); player.activeWeapon aponta para um objeto de WEAPON_DEFS com uma função draw. Funções não sobrevivem a JSON.stringify. A solução do SaveSystem é nunca serializar o objeto inteiro — apenas o id de cada coisa, e reconstruir a referência completa na hora de carregar, buscando de volta na tabela de definições original:

// Serializar: guarda só o id
const inventorySerial = p.inventory.map(it => ({ id: it.id, qty: it.qty ?? 1 }));
const equipmentSerial = {};
for (const [slot, item] of Object.entries(p.equipment)) {
  equipmentSerial[slot] = item ? item.id : null;
}

// Desserializar: reconstrói o objeto completo a partir do id
p.inventory = (data.inventory ?? []).map(saved => {
  const def = SHOP_ITEMS.find(s => s.id === saved.id) ??
              EQUIPMENT_ITEMS?.find?.(e => e.id === saved.id);
  return def ? { ...def, qty: saved.qty ?? 1 } : null;
}).filter(Boolean);

DarknessSystem: uma camada de sombra sobre um canvas próprio

Mapas escuros (cavernas, o santuário sombrio) não escurecem tile por tile — o sistema desenha uma camada preta sólida em um <canvas> off-screen próprio, depois "perfura" um círculo de luz suave ao redor do jogador (e de outras fontes de luz, quando existem) usando composição de canvas, e finalmente cola esse canvas por cima da cena principal:

class DarknessSystem {
  constructor() {
    this._canvas = document.createElement('canvas');   // camada off-screen
    this._ctx    = this._canvas.getContext('2d');
    this._flicker = 0; this._flickerTarget = 0; this._flickerT = 0;
  }

  update(dt, params) {
    if (!params?.flicker) return;
    // Flicker suave: interpola para um alvo aleatório periodicamente
    this._flickerT -= dt;
    if (this._flickerT <= 0) {
      this._flickerTarget = (Math.random()*2-1) * params.flicker;
      this._flickerT = 0.05 + Math.random()*0.1;
    }
    this._flicker += (this._flickerTarget - this._flicker) * Math.min(1, dt*12);
  }
}

RainSystem: a lição de câmera aprendida em produção

A sua memória de projeto registra um ciclo de depuração longo nesse sistema — colisão em world-space, alvos de queda aleatórios, compensação de deriva, artefatos de sincronização entre faixas de chuva. A abertura em dissolução de tiles

A animação de abertura do jogo — revelar o mundo a partir do centro da tela para fora, tile por tile, ao escolher um slot de save — usa a mesma ideia central de qualquer efeito de "dissolve" por grade: cada tile recebe um atraso de revelação calculado a partir da sua distância até o centro, mais um pequeno valor de aleatoriedade (jitter) por célula, para que a borda de revelação não seja um círculo perfeitamente liso.

Estendendo o Jogo

Três receitas práticas, seguindo exatamente os padrões dos capítulos anteriores.

Com a arquitetura mapeada, adicionar conteúdo novo deixa de ser "escrever um sistema" e passa a ser "preencher os mesmos moldes que já existem". Estas três receitas cobrem os casos mais comuns de expansão do jogo.

Receita 1 — Adicionar um novo mapa

  1. Escolha um id único e desenhe a matriz tiles (linhas de valores de T, capítulo 2). Lembre-se: as bordas por onde o jogador deve poder sair precisam ser tiles não-sólidos.
  2. Adicione a entrada em MAP_DEFS com tileW/tileH corretos e spawnDefault.
  3. Ligue o mapa novo aos vizinhos: adicione connections nele e na direção oposta do mapa vizinho (as conexões não são inferidas automaticamente — são declaradas nos dois lados).
  4. Povoe com enemies, items, npcs etc. usando coordenadas de tile (tx/ty), seguindo a mesma estrutura do exemplo de 'village' no capítulo 2.
  5. Se o mapa deve aparecer no mapa-múndi [M], adicione sua entrada em WORLD_LAYOUT com {gx, gy} relativo aos vizinhos, e uma aresta em WORLD_CONNECTIONS. Coordenadas negativas são seguras — o cálculo de offset do capítulo 3 já lida com isso.

Receita 2 — Adicionar um novo tipo de inimigo simples

Para um inimigo que só precisa de estatísticas diferentes (mais vida, mais velocidade, tabela de drop própria), nenhuma classe nova é necessária — basta uma entrada na tabela stats dentro do construtor de Enemy (capítulo 7):

const stats = {
  // ... existentes
  wolf: { hp:4, speed:95, color:'#a09070', dmg:1, range:260,
          dropTable:[{w:40,item:'gem'},{w:20,item:'heart'},{w:40,item:null}] },
};

Para um inimigo com comportamento novo (ataque à distância, fases, padrão de movimento especial), crie uma classe que estende Enemy (ou Boss, se tiver múltiplas fases), chame super.update(dt, player, map) para herdar movimento/melee, e adicione só a lógica extra — exatamente como Goblin e AngelBoss fazem no capítulo 7. Depois:

  1. Referencie o type novo em enemies: [...] de qualquer MAP_DEFS.
  2. Se o inimigo precisar ser instanciado como a classe especial (não a genérica Enemy), garanta que o código que lê def.enemies no carregamento do mapa saiba mapear aquele type para a nova classe.

Receita 3 — Adicionar uma nova arma corpo-a-corpo

Siga o molde de WEAPON_DEFS (capítulo 6): um novo objeto com id, name, unlockLevel, os três números de trade-off (attackDur, attackRange/spread, dmgMult) e uma função draw(ctx) desenhando a arma em pixel art simples (retângulos coloridos, o mesmo estilo de todas as armas existentes). Nenhum outro código precisa mudar — Player.attack() já lê os números da arma ativa dinamicamente, e a tela de armas [T] já itera WEAPON_DEFS automaticamente.

O fio condutor de todo o guia Em cada um dos treze capítulos, o mesmo princípio se repete sob nomes diferentes: comportamento como dado, orquestrado por um pequeno número de classes centrais — tiles em TILE_COLORS, armas em WEAPON_DEFS, itens com onUse, missões guiadas por QUEST_DEFS, mapas inteiros em MAP_DEFS. É esse padrão, mais do que qualquer classe específica, que faz um arquivo de 24 mil linhas continuar extensível sem virar um emaranhado. Ao adicionar algo novo, a primeira pergunta vale sempre a pena: "isso é uma classe nova, ou só uma linha nova numa tabela que já existe?"

Um raio-X didático sobre o próprio jogo: a mesma cena, redesenhada como diagrama — vetores, AABBs, distâncias e o caminho que a IA calculou pra te alcançar.

O Modo Matemática não é um sistema de jogo — é uma segunda função de desenho para a mesma cena.
